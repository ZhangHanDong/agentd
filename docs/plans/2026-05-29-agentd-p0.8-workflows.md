# P0.8 — Workflow Authoring: `draft.dot` + `execute.dot` + `agentctl run start` (Path-B Δ1)

> **For agentic workers:** this is a TDD plan. Each task is RED (failing test) → author/code → GREEN → gate → commit. agentd-core is FROZEN (D1) — do not edit any file under `crates/agentd-core/**` (including its `tests/`).

**Goal:** Ship the two standalone Path-B DOT workflows — `draft.dot` (issue → spec draft) and `execute.dot` (frozen spec → PR) — that conform to agentd-core's frozen DOT grammar and provably *walk* on the real `Engine` over fake ports, plus an `agentctl run start` CLI shell with a `--dry-run`/validate path. **Live cross-process execution is deferred to P0.9.**

**Architecture:** Pure authoring + validation + correctness-test phase. Touches only `workflows/`, `crates/agentctl/**`, and `specs/workflow/**`. The walk-tests live in `crates/agentctl/tests/` and construct the real `Engine` + the `agentd-core` `test-support` fakes (a dev-dependency) — so agentd-core stays untouched (D1). No production `RunHost`, daemon, rmcp transport, or live agent is built here; those are P0.9.

**Tech:** Rust 2024; `agentd-core` (frozen) `graph`/`engine`/`handler` + its `test-support` fakes; `agentctl` (clap); `agentctl flow validate` (existing). DOT = the Appendix-C dialect.

## Decisions

- **D1 — Scope = author + validate + dry-run; live execution → P0.9 (user-confirmed).** A complete `run start` that drives a workflow to `done` needs the P0.9-deferred production `RunHost` (cross-process unpark reconstructs `Engine<'a>` from checkpoint). This phase ships the two `.dot` files, the `open_pr` node, the standalone stub conventions, the `run start` CLI shell + `--dry-run`, and full walk-correctness tests. agentd-core frozen (D1).
- **D2 — Walk-tests use the REAL `Engine` + FAKE ports, in `crates/agentctl/tests/`.** `Engine::new` + `HandlerRegistry::with_builtins()` are `pub`; the fakes (`FakeStore`/`FakeBackend`/recording `CommandRunner`) ship in `agentd-core`'s `test-support` feature. agentctl gains a **dev-dependency** `agentd-core = { features = ["test-support"] }`. A test holds `Engine<'a>` across `execute()`→`deliver_event()` in one borrow scope, so a full in-process walk to `done` works today. **NOT `FakeRunHost`** — that scripts `RunProgress` and proves only the MCP-tool layer, not the engine walk.
- **D3 — DOT grammar conformance (frozen, `graph/node_graph.rs` + `graph/validate.rs`).** Handlers are exactly `codergen` / `conditional` / `tool` / `wait.human` / `parallel.fan_out` / `parallel.fan_in`. Shapes are case-sensitive: `shape=Mdiamond`→Start, `shape=Msquare`→Terminal; exactly one start, ≥1 terminal. Any `pre_tools`/`post_action` token must be in `KNOWN_TOOLS` (`assign_task, submit_review, check_inbox, submit_outcome, query_run, mempal_*, matrix.post, github.status_push`). Every edge endpoint must be a declared node; every non-start node forward-reachable; every `goal_gate=true` node must have a forward path to a terminal.
- **D4 — Static cmd strings only.** `tool` `cmd=` is whitespace-split into program + argv (`handler/tool.rs`); **no quote handling, no `${...}`/`$run_dir` substitution**, and `RunOpts.cwd`/`env`/`stdin` are NOT populated (only `timeout`). Per-run value injection is a P0.9 host responsibility. Authored cmds must be static and must not depend on cwd/env.
- **D5 — `goal_gate_unmet` recovery edges (runtime-only; `validate.rs` does NOT check this).** `evaluate_goal_gate` is global (all `goal_gate=true` nodes) and fires when a transition targets a terminal; if unmet it synthesizes a `goal_gate_unmet` Fail and re-selects ONCE (`engine/execute.rs:390-414`). So the node whose edge routes to `done` MUST have a recovery edge `[label="goal_gate_unmet"]` to a **non-terminal that can satisfy the gate on re-run** (route to `implement`, NOT an un-gated self-loop → that hits STEP_CEILING). `outcome=goal_gate_unmet` in a `condition=` also matches (the `outcome=` atom checks `preferred_label`).
- **D6 — `open_pr` = `tool` node (Q3 default: ambient gh auth).** `handler=tool, cmd="gh pr create --fill --head ..."` (static argv), `timeout_secs=`. In standalone P0 the token comes from the operator's ambient `gh` auth / `GH_TOKEN` in the run env (documented requirement); Specify-provisioned injection is P0.9/P1. `post_action="matrix.post(...)"` validates against KNOWN_TOOLS but is a **runtime no-op** in core (no executor) — label it; PR notifications wire in P0.9.
- **D7 — Standalone freeze = local file (Q2 default).** Between `draft` and `execute`, the frozen spec is a local file (e.g. `pull_frozen_spec` is a `cat`/`cp` `tool` node over a fixed standalone path); no git-tag coupling, no HTTP to Specify. Standalone conventions stay thin cmd-string conventions behind a config/env flag, easy to swap for the P1 `agentd-specify` client. Cross-run draft→execute artifact linkage is a noted limitation (ephemeral runs; the frozen-spec path is passed explicitly), not a blocker.
- **D8 — `run start --flow {draft|execute}` (Q4 default).** The flag selects the `.dot`; `--dry-run` parses + validates + prints the resolved plan (NO live execution); the live path returns a clear "live execution deferred to P0.9 (no daemon)" error — no silent hang. Unknown flow / missing file → clean error.
- **D9 — `install-skills` (the original P0.8 other half) is NARROWED out / deferred.** This phase is the Path-B workflow-authoring slice. `agentctl install-skills` + the shipped-skills concern is tracked separately (a P0.8 follow-up or P0.9 tooling), not in this tag.

- RED: a test asserting `workflows/draft.dot` parses + `NodeGraph::from_ast` validates (fails — file absent).
- Author `draft.dot`: `start`(Mdiamond) → `fetch_issue_context`(tool, static cmd) → `propose_spec`(codergen, role=spec-writer, initial_prompt_includes=…) → `lint_spec`(tool, `cmd="agent-spec lint …"`) → `push_draft`(tool, static cmd) → `done`(Msquare). Edges chain them. GREEN.
- Gate (validate + lint) + commit.

### Task 2 — `draft.dot` walk-test (real Engine + fake ports)
- **Files:** `crates/agentctl/tests/workflows.rs`, `crates/agentctl/Cargo.toml`
- Add the `agentd-core` `test-support` dev-dep. Build a small in-test harness (real `Engine::new` + `HandlerRegistry::with_builtins()` + fake `Store`/`Backend`/`CommandRunner`).
- RED→GREEN: `execute()` walks `start`→`fetch_issue_context`(tool runs)→ parks at `propose_spec` (codergen Park); `deliver_event(AgentOutcomeSubmitted success)` resumes → `lint_spec` → `push_draft` → `done`; assert final run status Finished and the path order.
- Gate + commit.

### Task 3 — `execute.dot`: spec + validate, incl. goal_gate recovery edges (RED→GREEN)
- **Files:** `specs/workflow/p81-execute-dot.spec.md`, `crates/agentctl/tests/workflows.rs`, `workflows/execute.dot`
- Author **p81** (scenarios: validates; walks to done; review fan_out collects N then fan_in aggregates; **goal_gate unmet routes to recovery, not Stuck**; an execute.dot missing a recovery edge would get Stuck — documented). Lint ≥ 0.7.
- RED: validate test for `execute.dot`.
- Author `execute.dot`: `start` → `pull_frozen_spec`(tool, `cat` the local frozen file) → `draft_plan`(tool, `agent-spec plan`) → `implement`(codergen, role=implementer) → `verify_lifecycle`(tool, goal_gate=true) → `review`(parallel.fan_out, reviewers="claude-sec,codex-perf,gemini-readability", bundle=frozen, visibility=blind) → `aggregate`(parallel.fan_in, aggregator=majority_pass, goal_gate=true) → `open_pr`(tool, `gh pr create …`) → `report_acceptance`(tool) → `done`. Plus the **recovery edge** `report_acceptance -> implement [label="goal_gate_unmet"]` and `report_acceptance -> done [condition="outcome=success"]`. GREEN validate.
- Gate + commit.

### Task 4 — `execute.dot` happy-path walk-test
- **Files:** `crates/agentctl/tests/workflows.rs`
- RED→GREEN: walk `execute.dot` with all gates satisfied: park at `implement`→resume success; `verify_lifecycle` success; `review` fan_out spawns 3 → `deliver_event(ReviewVerdictSubmitted ×3 pass)`; `aggregate` majority_pass success; `open_pr` tool runs (assert the recording `CommandRunner` saw `gh pr create` as program+argv per D4); `report_acceptance`; gate MET at the `done` transition → Finished.
- Gate + commit.

### Task 5 — `execute.dot` NEGATIVE goal_gate walk-test (the signature failure mode)
- **Files:** `crates/agentctl/tests/workflows.rs`
- RED→GREEN: drive the walk so a `goal_gate=true` node is UNMET (e.g. `deliver_event` a Fail for `verify_lifecycle`, or never satisfy it) yet the run still reaches `report_acceptance`. Assert the `report_acceptance`→`done` transition is gated: the engine re-selects the `[label="goal_gate_unmet"]` recovery edge → routes to `implement` (non-terminal), **NOT Stuck/Failed**. (Guards D5 — a missing/mis-conditioned recovery edge would Stuck or, if self-looping un-gated, hit STEP_CEILING.)
- Gate + commit.

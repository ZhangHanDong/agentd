# P0.7 — agentd-surface: rmcp MCP server + HTTP/SSE (Path-B next phase)

> Generated at phase start from design §1.2/§4.12.1/§7.2. This is the standalone-MVP
> critical path: agents in tmux panes reach the workflow engine through the 5 MCP
> tools. Split into **7a** (MCP server + tools — do first) and **7b** (HTTP+SSE +
> events persistence). Every tool handler is tested against in-memory ports; the
> rmcp/axum transport is thin and smoke-tested — no real agent/network in v0.

## Architecture decisions (locked before coding)

- **D1 — Zero edits to the tagged `agentd-core` (invariant held since P0.3).**
  Orientation reality: the engine is `Engine<'a>` — a **per-call, borrow-based**
  struct (`Engine::new(&graph, &registry, Ports{&backend,&store,&mempal,&clock})`
  then `execute`/`deliver_event`), and `agentd-bin` is still a P0.0 stub (no
  daemon orchestrator owns the engine pieces yet). So the surface tools cannot
  "hold an Engine". Instead `agentd-surface` defines a **`RunHost` seam** (the
  agent↔engine/store boundary) and the 5 tools are pure functions over
  `Arc<dyn RunHost>`:
  - `deliver(EngineEvent) -> RunProgress` (writes: submit_outcome / submit_review)
  - `run_snapshot(run_id) -> Option<RunSnapshot>` (query_run)
  - `open_task(run_id, node_id) -> Option<TaskAssignment>` (assign_task + the submit_outcome `task_run_id` resolution)
  - `review_counts(review_run_id) -> (expected, got)` (submit_review `fan_in_pending`)
  The PRODUCTION `RunHost` (constructs `Engine` with the real ports + the run's
  graph + reads the store/checkpoint) is the **daemon's** job, **deferred to
  daemon-wiring (P0.9)** — same "defer real wiring" call as the rmcp transport (D7).
- **D2 — Seam-tested, like P0.3/P0.4.** The 5 tools are tested against a
  `FakeRunHost` (records delivered events, returns scripted `RunProgress` /
  snapshots / task assignments). The rmcp server (stdio) and axum router are thin
  transport wrappers, smoke-tested (server builds, `tools/list` lists 5; routes via
  `tower::ServiceExt::oneshot`). No real engine, MCP client, or socket in v0. The
  real engine integration is already covered by core's `engine_execute.rs`
  (`deliver_event` with `AgentOutcomeSubmitted`/`ReviewVerdictSubmitted`) and the
  end-to-end wiring lands in P0.9.
- **D3 — Tool → `RunHost` mapping (§4.12.1).** Each tool is a pure function over
  `Arc<dyn RunHost>` (the fake scripts these in tests; the real daemon host does
  the engine/store work):
  - `submit_outcome` → `host.open_task(run, node)` (resolve `task_run_id`; the inputs carry only `(run,node,attempt)` while `deliver` routes by `task_run_id`) → build `Outcome` + `host.deliver(AgentOutcomeSubmitted{task_run_id, outcome})`. Codes: `open_task` miss → `not_found`/`not_assigned`; `RunProgress::Ignored` (park moved) → `stale_attempt` — do NOT collapse the two `Ignored` causes.
  - `submit_review` → `host.deliver(ReviewVerdictSubmitted{…})`; `accepted` from the result; `fan_in_pending` from `host.review_counts` (= expected − got). Conflicting re-submit → `already_submitted`.
  - `query_run` → `host.run_snapshot(run)` → `{status, current_node, completed_nodes, context}`; `None` → `not_found`. (The real host reads run_repo + the checkpoint, since the core `Store` trait has no run-state read; D1-clean store reads.)
  - `assign_task` → `host.open_task(run, node)` (+ verify `agent_id`) → `{task_run_id, worktree, spec_path?, plan_path?, context_pack?}`; miss → `not_assigned`.
  - The real host's `open_task` adds an agentd-store `task_repo::find_open_task_run(run, node)` read (the store only has the reverse `lookup_park_by_task_run`) — that lands when the production host is wired (P0.9), not in 7a.
- **D4 — Errors as MCP codes.** A private `SurfaceError` maps to the §4.12.1 codes (`not_assigned` / `already_submitted` / `stale_attempt` / `not_found`) in the tool result, and to HTTP status on the SSE/HTTP side.
- **D5 — `check_inbox` does NOT widen the frozen `MempalClient` port (FLAG).** The
  cowork-bus *pull* isn't on the port (only search/ingest/kg_add/fact_check). v0
  routes the pull through the **mempal transport seam** (`McpToolCaller`, a
  cowork-bus pull tool) from `agentd-mempal`, OR returns an **empty inbox**
  (documented degraded mode) — the standalone MVP tolerates no peer messages.
  Either way, agentd-core stays frozen. *Confirm which before 7a's check_inbox task.*
- **D6 — Events persistence is NEW work in 7b (extends agentd-store, FLAG).** The
  `events` table (P0.2) exists but nothing writes/reads it. 7b adds an
  `event_repo` (append + read-from-seq) and a thin emit point (the daemon/executor
  records an event row per `RunProgress`/`EngineEvent`); the SSE endpoint replays
  from `?from_seq=` over `idx_events_run_seq`. The `EngineEvent`→event-row mapping
  is defined here (Δ8's Specify-schema serialization stays P1).
- **D7 — The real rmcp CLIENT transport (mempal, deferred from P0.4) stays deferred (FLAG).**
  P0.7 builds the agentd-as-MCP-**server** side. The mempal client seam remains
  fake-tested; a real mempal connection is not on the standalone-MVP critical path
  (the drainer is best-effort, mempal-down is tolerated). Land it in P0.9 E2E.
- **D8 — Split 7a then 7b.** 7a (MCP server + 5 tools) is the agent-execution
  critical path; 7b (HTTP+SSE + events) is dashboard/observability. Tag
  `v0.0.0-p0.7` after 7b (or `v0.0.0-p0.7a` / `-p0.7b` if you prefer per-half tags).
- **D9 — Known P0.9 hazard: SSE field sanitization (FLAG, surfaced by the P0.7 adversarial review).**
  `http::run_events` builds `Sse` frames via `Event::data(payload)` / `Event::event(kind)`
  from the opaque `EventRecord`. axum 0.7's `Event::data`/`::event` **panic on a CR/LF**
  in the value (reproduced through the real router via `tower::oneshot`). This is **not**
  reachable in P0.7 — there is no emit point, so only test-controlled payloads reach the
  handler — but it becomes live with the P0.9 **emit point**. When wiring emit: serialize
  event payloads **compactly** (`serde_json::to_string`, no pretty-print newlines) and/or
  sanitize `kind`/`payload` at the SSE boundary, and add a `payload-with-newline` scenario
  to p73. Do not let the emit wiring ship without it.

## Task breakdown (each: spec → failing test → impl → clippy/fmt → `./scripts/check.sh` → `cargo deny` → commit `-F`)

### 7a
- **Task 1 — surface skeleton + state + `query_run` + `submit_outcome`.** `state.rs` holds the injected `EngineExecutor`/`Store`/`MempalClient`; `error.rs` (`SurfaceError`→codes). Tools `query_run` (store read; `not_found`) and `submit_outcome` (→ `AgentOutcomeSubmitted` → `deliver_event`; append-once → `stale_attempt`). Spec **p70** (schemas) + **p72** (idempotency). Tests against a real engine + `InMemoryStore`/`FakeBackend`.
- **Task 2 — `submit_review` + `assign_task`.** review → `ReviewVerdictSubmitted` (`{accepted, fan_in_pending}`, `already_submitted`); assign_task → parked task_run read (`not_assigned`). Spec **p74** (error codes). Tests drive the engine to a parked review/task and exercise both.
- **Task 3 — `check_inbox` + the rmcp MCP server wiring.** check_inbox per **D5** (confirm route). `mcp_server.rs` registers the 5 tools on an rmcp stdio server; smoke test: server builds, `tools/list` returns 5, a `tools/call` round-trips to a handler. Fold tools into `tools/mod.rs`.

### 7b
- **Task 4 — `event_repo` + emit.** `event_repo::append`/`read_from(run_id, seq)`; emit an event row per `RunProgress`/`EngineEvent` at the daemon seam. Spec **p71** (replay).
- **Task 5 — axum HTTP routes + SSE replay.** `http.rs` routes (`GET /runs/:id`, `GET /healthz`, …) + `sse.rs` (`GET /runs/:id/events?from_seq=`) replaying from the `events` table. Spec **p73** (routes). Tests via `oneshot`.
- **Task 6 — acceptance sweep + adversarial review + tag** `v0.0.0-p0.7`; roadmap row → done.

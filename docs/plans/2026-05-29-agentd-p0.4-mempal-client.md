# P0.4 — mempal MCP Client + Outbox Drainer + Consistency Check

> Generated at phase start (roadmap policy) from design §3.1/§3.4/§3.5/§4.12.2.
> Implements the mempal integration: a best-effort MCP client behind an injected
> transport seam (no real rmcp/mempal server — every test runs against a fake),
> the transactional outbox enqueue + FIFO drainer, and a read-only consistency
> check. Mempal-down must never stall a workflow.

## Architecture decisions (locked before coding)

These resolve the cross-crate seams up front (advisor- and operator-reviewed):

- **D1 — The outbox ENQUEUE lives in `agentd-store`'s `outcome_repo`, in the same
  DB transaction as the `node_outcome` insert (design §3.4).** `agentd-core`
  already carries `Outcome.mempal_writes: Vec<MempalWrite>` and the
  `outcome_repo` header already states these "flow to `mempal_outbox`" — P0.4
  wires that: one `MempalWrite` → one `mempal_outbox` row (`kind` = the op,
  `payload` = the write's JSON) committed atomically with the outcome. This
  **extends the tagged `agentd-store`** (the designed path; not a frozen-core
  situation). **Zero edits to `agentd-core`** — the `MempalClient` port,
  `MempalWrite`, `DrawerHit`, and `MempalStub` already exist there.

- **D2 — The MCP client, drainer, and consistency check live in `agentd-mempal`**,
  which depends on `agentd-core` (port + types) and `agentd-store` (outbox repo +
  pool). The drainer is the only component that calls the real mempal.

- **D3 — Inject the MCP transport as an object-safe seam
  (`Arc<dyn McpToolCaller>`), tested against a fake — no real rmcp/mempal server
  in v0 (operator decision).** `MempalMcpClient` implements the core
  `MempalClient` port by mapping each method → a mempal MCP tool call
  (`mempal_search` / `mempal_ingest` / `mempal_kg add` / `mempal_fact_check`,
  §4.12.2) through the transport. Tests inject a `RecordingToolCaller` that
  records the tool name + JSON args and replays scripted responses. The real
  rmcp-backed transport is **deferred to P0.7** (the MCP phase) and recorded as a
  known follow-up; v0 ships the seam + a documented `TODO`-free deferral note.

- **D4 — The client HONORS the port contract; the empty-fallback is the caller's
  and already exists (verified in core).** The `MempalClient` port doc says
  `search`/`fact_check` *return* `CoreError::Mempal` when mempal is unreachable,
  and `codergen.rs` already calls `.search(...).await.unwrap_or_default()` — i.e.
  the engine handler is the pre_tools execution site that "proceeds with empty"
  (§3.4). So `MempalMcpClient` wraps reads in a `Config` timeout (3s default) and
  maps a timeout/transport failure to `Err(CoreError::Mempal)` **+ a warning
  log** — it does NOT swallow to empty (that would break the contract and diverge
  from `MempalStub`). P0.4 does not duplicate the fallback; the client's job is to
  *time out to a clean `Err` rather than hang*. Timeout is `Config`-driven so
  tests zero it.

- **D5 — Error type is `CoreError`, NOT `MempalError`, at the boundary.** Same
  move as P0.2/P0.3: `agentd-mempal` keeps a private `MempalError` (transport /
  timeout / decode variants) and maps to `CoreError::Mempal(String)` where a
  public surface returns `CoreError` (the `MempalClient` impl). Internal helpers
  return `MempalError`.

- **D6 — The drainer is a `tokio::spawn` background loop** (design §3.4): claim
  pending rows (`drained_at IS NULL`, FIFO `enqueued_at`), dispatch each to the
  `MempalClient`, set `drained_at` on success; on failure bump `attempts`, record
  `last_error`, and back off exponentially. Rows with `attempts > 5` stay in the
  table and emit an operator alert (`tracing::error!` + an observable
  `DrainReport`). Enqueue is fire-and-forget inside the outcome tx, so a slow or
  down mempal never blocks workflow execution.

- **D7 — `palace.db` is the forbidden token this phase (design §3.1).** agentd
  never `INSERT`s into mempal's DB; MCP is the only channel. `check.sh` layer 7
  already greps `crates/*/src/**` for `palace.db`; no `agentd-mempal` source may
  reference it. Specs `Forbidden`-clause that literal only.

## Task breakdown (each: spec → failing test → impl → clippy/fmt → `./scripts/check.sh` → `cargo deny --all-features check` → commit `-F`)

### Task 1 — agentd-mempal skeleton: error, McpToolCaller seam, MempalMcpClient, pre_tools best-effort
- `error.rs`: `MempalError` (Transport / Timeout / Decode) + `From<MempalError> for CoreError` (→ `CoreError::Mempal`).
- `transport.rs`: `McpToolCaller` (`async fn call_tool(&self, tool: &str, args: serde_json::Value) -> Result<serde_json::Value, MempalError>`), object-safe; production rmcp impl deferred to P0.7 (documented).
- `config.rs` (or inline in client): `pre_tools_timeout` (3s default), overridable/zeroable in tests.
- `client.rs`: `MempalMcpClient { caller: Arc<dyn McpToolCaller>, cfg }` implementing `agentd_core::ports::MempalClient`; map `search`→`mempal_search`, `ingest`→`mempal_ingest`, `kg_add`→`mempal_kg` (op=add), `fact_check`→`mempal_fact_check`; parse `{hits|issues}` → `Vec<DrawerHit>`; reads (`search`/`fact_check`) wrapped in the `pre_tools_timeout` and mapping timeout/transport failure to `Err(CoreError::Mempal)` + warn (D4 — NOT empty); writes map errors to `CoreError::Mempal`.
- `test_support.rs`: `RecordingToolCaller` (records `(tool, args)`, replays scripted `Result`s FIFO; a `hang` mode returns a genuinely never-resolving `std::future::pending()` so the zeroed-`Config` timeout actually trips — an instant return or `sleep` would not).
- Specs **p32** (client-mcp-tools) + **p33** (pre-tools-best-effort).
- Tests (`tests/client.rs`): each port method issues the right tool name + args; `search`/`fact_check` parse hits; **`test_pre_tools_search_falls_back_to_empty_on_timeout`** = with the hanging caller + zeroed timeout, `client.search(...)` returns `Err(CoreError::Mempal)` *within the timeout (never hangs)*, and a caller's `.unwrap_or_default()` on it therefore yields `vec![]` (mirroring `codergen.rs`, which owns the actual fallback).

### Task 2 — outbox repo + transactional enqueue (agentd-store)
- **First verify the write path:** confirm the `insert_outcome` signature actually receives a populated `Outcome.mempal_writes` (the `outcome_repo` header only documents the *read* path reconstructing empty writes). If the insert drops the writes, wiring it to carry them is part of this task (still agentd-store, D1-clean).
- `outbox_repo.rs`: `enqueue(tx, run_id, node_id, writes)` (one row per `MempalWrite`, `enqueued_at` = clock), `claim_pending(limit)` (FIFO `enqueued_at`, `drained_at IS NULL`), `mark_drained(id, at)`, `mark_failed(id, last_error, at)` (bump `attempts`).
- `outcome_repo.rs`: within the existing `insert_outcome` tx, also `enqueue` the outcome's `mempal_writes` (atomic with the `node_outcome` row).
- Spec **p30** (outbox-drainer, enqueue half).
- Tests (`tests/outbox.rs`): **`test_kg_add_writes_outbox_row_in_same_tx_as_node_outcome`** — prove ATOMICITY by ROLLBACK, not co-presence: drive the outcome insert into a tx abort (e.g. a `KgAdd` outcome whose `run_id` violates the FK, or a forced failure after the outcome row), then assert **neither** the `node_outcome` nor the `mempal_outbox` row persisted; and a success path leaves exactly one outbox row per write. **`test_ingest_via_outbox_does_not_block_workflow`** (enqueue returns without any mempal call; the row is pending, `drained_at IS NULL`).

### Task 3 — drainer (background FIFO + exp backoff + alert)
- `drainer.rs`: `drain_once(store, client) -> DrainReport` (claim pending FIFO, dispatch each `MempalWrite` to the `MempalClient`, `mark_drained` on `Ok`, else `mark_failed` + backoff; `attempts > 5` → leave + alert), and `spawn(store, client, cfg)` running `drain_once` on an interval with exponential backoff between empty/erroring passes.
- Spec **p30** (drainer half).
- Tests (`tests/drainer.rs`): a pending `Ingest` row drains and is marked `drained_at` (FIFO order across two rows); **`test_drainer_retries_with_backoff_until_attempts_exceeded`** (erroring client → `attempts` climbs to 6, row stays pending, `DrainReport` flags the alert); a down mempal leaves the workflow untouched.

### Task 4 — consistency check
- `consistency.rs`: `check_against(expected: &[ExpectedDrawer], client) -> Vec<ExpectedDrawer>` — `mempal_search` per expected drawer (drained `Ingest` writes agentd believes it pushed); return those mempal does **not** return. Read-only; reports drift, does not re-ingest (git wins, §3.5).
- Spec **p31** (consistency-check).
- Tests (`tests/consistency.rs`): **`test_consistency_check_reports_missing_drawers`** (script the client to return some expected drawers and omit others → the omitted ones are reported; all-present → empty report).

### Task 5 — acceptance sweep + adversarial review + tag
- Confirm the five roadmap tests + all spec scenarios; `cargo test -p agentd-mempal -p agentd-store --no-fail-fast` green.
- Run the adversarial-verification workflow over the outbox-tx atomicity, the drainer backoff/alert + FIFO, and the pre_tools best-effort invariants.
- Fold findings; final gate (`check.sh` + `cargo deny` + `cargo build --locked`); tag `v0.0.0-p0.4`; roadmap row → done.

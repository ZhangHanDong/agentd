# P0.9 Deployment Checklist — the real-environment capstone

P0.9 was completed **code-complete + contract-tested in-process** (see
[`2026-05-29-agentd-p0.9-capstone.md`](./plans/2026-05-29-agentd-p0.9-capstone.md)).
The offline build box has **no real AI agents, no mempal server, no tmux session,
no docker/synapse, and cannot SIGKILL itself mid-assertion**, so the items below —
everything that needs a real process boundary or external service — are delivered
as a runnable checklist to execute on a real machine. None of them is new logic;
each exercises an already-tested seam against a real counterpart.

Legend: **[proven-by-test]** = the contract is verified in-process; this step adds
only environment coverage.

- [ ] Re-launch the daemon with the **same** `--db-path`. ## Known gaps to close in the real-env wiring (not P0 blockers)

- **TaskAssignment** (P0.9 D5): P121 resolves the old `agent_id` gap by persisting task-run ownership in `task_runs.agent_id` and returning it from production `open_task`; P136 resolves the runtime metadata bridge by returning checkpoint context strings for `spec_path`/`plan_path`/`context_pack` when present. These values still come from workflow context producers such as P127's `execute.dot` staging, not from new schema columns.
- **Initial run context** (P0.9 9c): P137 resolves the production seeding bridge: `start_workflow` now seeds JSON object `context` into the initial `RunContext` (JSON null stays empty; non-object values are rejected before start). Real runs can pass `issue_id`/`spec_path` through `POST /runs` JSON or `agentctl run start --context-file`, while fixed `.agentd/run/*` files remain the standalone Path-B convention.
- **Checkpoint/outcome atomicity** (advisor/P1): ✅ RESOLVED (P138): completed-node outcomes and their following checkpoints now commit through one Store operation backed by a single SQLite transaction; `mempal_outbox` enqueue remains in that same transaction.
- **SSE field sanitization** (P0.7 D9): ✅ RESOLVED (P139): the SSE boundary sanitizes CR/LF in event names and payload data before calling axum; persisted `EventRecord` rows and production emit payload JSON shapes remain unchanged.
- **Re-park emit noise** (P1) — ✅ RESOLVED (`6f41188`): the emit fired on every state-changing `RunProgress`, so a re-park at the SAME node (e.g. a fan-out review collecting verdicts) emitted a fresh `run_parked` per non-final verdict — truthful but noisy for an SSE dashboard. Fixed by deduping same-node re-parks at the emit point (suppress when the run's most recent event is already a `run_parked` for that node — `event_repo::last`); the `goal_gate_unmet → implement` re-loop still emits because a `review` park intervenes. See `specs/e2e/p90-runhost-contract.spec.md`.
